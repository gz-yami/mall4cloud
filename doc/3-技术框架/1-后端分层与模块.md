# 后端分层与模块

Mall4cloud 后端是多模块 Maven 工程。二开时优先沿用现有分层，不要为了一个功能引入新架构。

## 常见分层

| 层 | 常见目录 | 职责 |
| --- | --- | --- |
| Controller | `controller` | 接收 HTTP 请求、做参数校验、调用 Service、返回 `ServerResponseEntity` |
| DTO | `dto` | 前端传入参数 |
| VO | `vo` | 返回给前端的数据 |
| Service | `service`、`service/impl` | 业务编排、事务边界、跨服务调用 |
| Mapper | `mapper` | MyBatis 数据访问接口 |
| XML | `resources/mapper` | SQL |
| Model | `model` | 与数据库表结构对应 |
| FeignClient | `mall4cloud-api-*` | 服务间调用契约 |
| Common | `mall4cloud-common-*` | 公共响应、异常、分页、缓存、安全、消息等能力 |

## 一个后台列表请求怎么走

以常见列表页为例：

1. 前端页面调用 `src/api/**` 中的方法。
2. 请求进入网关 `mall4cloud-gateway:8000`。
3. 网关按 Nacos 路由转发到对应服务。
4. Controller 接收 `PageDTO` 和搜索条件。
5. Service 调 Mapper 查询。
6. `PageUtil.doPage` 生成 `PageVO`。
7. Controller 返回 `ServerResponseEntity.success(pageVO)`。

## Feign 的位置

跨服务调用不要直接拼 URL。项目把内部接口放在 `mall4cloud-api` 下，例如商品库存锁定、用户信息、权限校验等。服务实现方实现对应 Feign 接口，调用方只依赖 API 模块。

## 新功能落点

- 后台 CRUD 功能优先找同业务域已有的 `HotSearch`、`IndexImg`、`Attr`、`Category` 模式。
- 需要跨服务时，先判断是否已有 `mall4cloud-api-*` 接口；没有再补接口契约。
- 需要事务跨服务时，参考已有 `@GlobalTransactional` 用法，不要只加本地事务。

## 检查题

问题：新增后台表单时，前端传来的对象应该直接用 Model 接收吗？

答案：不要，优先使用 DTO。

为什么：当前项目用 DTO 承接前端输入，用 Model 对应数据库表，用 VO 返回前端。直接暴露 Model 容易把数据库字段和接口契约绑死。
