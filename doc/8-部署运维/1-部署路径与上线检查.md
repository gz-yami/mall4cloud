# 部署路径与上线检查

部署前先区分三种路径。

## 路径一：本地开发

适合二开：

1. 本地或服务器启动中间件。
2. IDEA 启动后端服务。
3. `pnpm dev` 启动平台端或商家端。
4. 前端 `VITE_APP_BASE_API` 指向网关 `8000`。

## 路径二：单机服务器中间件 + 本地代码

适合团队共享中间件：

1. 服务器用 [中间件 Docker Compose](./2-中间件-docker-compose/README.md) 启动 MySQL、Redis、Nacos 等。
2. 本地服务通过 `NACOS_HOST` 指向服务器。
3. 前端资源地址指向服务器 MinIO。

## 路径三：生产部署

生产部署需要额外确认：

- Nacos、Redis、MySQL、MinIO、RocketMQ、Elasticsearch 不应直接暴露公网。
- RocketMQ dashboard 当前未配置登录保护，不能裸露。
- 前端构建后由 Nginx 提供静态资源。
- 前端 API 地址要指向生产网关域名。
- MinIO 桶策略和资源域名要按生产安全要求配置。

## Nginx 注意点

平台端和商家端当前仓库内的 `nginx.conf` 只配置了静态资源根目录，没有代理后端 API。生产环境更常见做法是：

- 静态资源由 Nginx 服务。
- API 通过 `VITE_APP_BASE_API` 指向网关域名。
- 或在外层 Nginx 单独配置 API 反向代理。

不要假设前端工程里的 Nginx 配置已经覆盖生产代理。

## 上线检查

| 检查项 | 正确结果 |
| --- | --- |
| 网关 | `8000` 或生产域名可访问 |
| Nacos | 服务已注册，配置可读取 |
| MySQL | 业务库、Nacos 库、Seata 库存在 |
| Redis | 密码和 Nacos 配置一致 |
| MinIO | 桶存在，资源地址可访问 |
| 前端 | `VITE_APP_BASE_API` 指向网关 |
| 登录 | 平台端或商家端能登录 |
| 权限 | 普通角色权限符合预期 |
| 日志 | 后端错误能在对应服务日志中定位 |

## 检查题

问题：生产环境只改前端 `.env.production` 就够了吗？

答案：不够。

为什么：还要核对 Nacos 中数据源、Redis、OSS、RocketMQ、Seata、网关路由，以及中间件自身的网络和安全暴露。
