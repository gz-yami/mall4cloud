# 启动、登录、接口常见问题

先按现象排查，不要一上来改代码。

## 服务连不上 Nacos

答案：先核对 `NACOS_HOST`、`NACOS_PORT`，再看 Nacos 容器、控制台和客户端端口。

为什么：各服务默认 Nacos 地址是 `${NACOS_HOST:192.168.1.46}:${NACOS_PORT:8848}`，如果你的环境不是这个 IP，就要改环境变量或启动配置。当前 Compose 的 Nacos v3 控制台是 `8080`，服务客户端主要看 `8848`，同时要放开 `9848`、`9849`。

排查顺序：

1. 在服务器执行 `docker ps -a`，确认 `mall4cloud-nacos` 没有反复重启。
2. 用 `docker logs -f mall4cloud-nacos` 看是否 MySQL 连接、账号或初始化库错误。
3. 打开 `http://<IP>:8080/index.html`，确认能登录控制台。
4. 在配置列表里确认 `application.yml`、`mall4cloud-gateway.yml` 等配置存在。
5. 如果 Nacos 配置已改但服务仍读到缓存地址，重启 Nacos 容器后再重启 Java 服务。

## 前端登录请求 404

答案：先核对 `VITE_APP_BASE_API` 是否指向网关 `8000`，不是某个业务服务端口。

为什么：前端接口带 `/mall4cloud_auth`、`/mall4cloud_rbac` 等前缀，网关负责按前缀转发。

## 前端请求 503

答案：先查网关路由目标服务是否启动并注册到 Nacos。

为什么：503 常见于网关能收到请求，但后面的 auth、rbac、platform、multishop、biz 等服务没有可用实例。

## 前端登录返回未授权

答案：检查账号密码、auth 服务、Redis、数据库初始化是否一致。

为什么：登录依赖 `mall4cloud_auth.auth_account`，Token 校验依赖 auth 服务和 Redis。

## 滑动验证码加载或校验失败

答案：先用浏览器 F12 的 Network 看验证码接口请求，再查 Redis。

为什么：验证码校验依赖 Redis。Redis 未启动、密码不一致，或 Nacos `application.yml` 中 Redis 地址仍指向错误 IP，都会导致验证码正常显示但校验失败。

## 登录后菜单不显示

答案：先判断用户是否管理员；非管理员再查角色是否拥有菜单。

为什么：非管理员会通过 `list_menu_ids` 获取菜单 ID，前端只显示这些 ID 对应的路由。

## 按钮不显示

答案：查按钮上的 `v-permission` 编码是否存在于用户权限列表。

为什么：前端 `src/directive/permission` 会从用户 store 的 `roles` 里查编码，没有就移除按钮。

## 接口 401 或权限失败

答案：查 `Authorization` 请求头、Token 是否过期、`menu_permission.uri` 和 `method` 是否匹配。

为什么：后端 `AuthFilter` 先校验 Token，再校验平台端/商家端 RBAC 权限。

## 图片不显示

答案：核对 MinIO 桶、桶策略、Nacos `biz.oss.resources-url`、前端 `VITE_APP_RESOURCES_URL`。

为什么：上传、返回路径、前端拼接地址、桶读取权限任何一处错都会导致图片不可见。

## 商品发布后搜索不到

答案：除 MySQL 外，还要查 Canal、RocketMQ、Elasticsearch。

为什么：搜索列表依赖 ES 索引，索引更新链路不等于商品表写入成功。

排查顺序：

1. MySQL 执行 `SHOW MASTER STATUS`，确认 binlog 可读。
2. 检查 `canal/conf/example/instance.properties` 中 MySQL 地址、`mysql-binlog.*` 文件名、`canal` 账号密码。
3. 检查 `canal/conf/canal.properties` 中 `rocketmq.namesrv.addr`。
4. 看 `canal/logs/example/example.log` 是否还在报错。
5. 看 RocketMQ dashboard 是否收到 `canal-topic`。
6. 看 `mall4cloud-search` 是否消费 `canal-topic`，ES 中是否已创建 `product`、`order` mapping。

## RocketMQ 无法发送 topic

答案：遇到 `No route info of this topic: stock-unlock-topic`，先查 RocketMQ broker 配置挂载和目录权限。

为什么：一种常见情况是 broker 没读到挂载的配置，导致 topic 路由不符合项目预期。当前中间件目录要求启动前给 `rocketmq/` 目录足够权限。

## Seata 连接失败

答案：先核对 `seata/application.yml` 的 MySQL 地址和 `application.yml` 中 `seata.service.grouplist.default`。

为什么：当前 Seata 2.6 compose 只暴露 `8091`，本仓库没有使用 `7091` 控制台配置。

## 检查题

问题：跨域报错时，第一反应是改业务 Controller 吗？

答案：不是，先查网关和前端 API 地址。

为什么：跨域主要发生在浏览器访问网关时，Nacos 中网关配置包含 CORS 规则，前端也必须请求正确网关地址。
