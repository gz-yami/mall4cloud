# 文件上传与富文本安全

当前文件上传主要由 `mall4cloud-biz` 的 OSS/MinIO 能力承接，富文本展示在前端通过 DOMPurify 做清洗。

## MinIO 上传链路

核心类：

- `OssController`
- `OssConfig`
- `MinioTemplate`

常见流程：

1. 前端请求 `/mall4cloud_biz/oss/info?fileNum=...` 获取上传地址。
2. 后端根据 `biz.oss.*` 配置创建 MinIO 预签名上传 URL。
3. 前端上传到 MinIO。
4. 前端展示图片时使用 `VITE_APP_RESOURCES_URL` 拼接资源路径。

也存在 `/oss/upload_minio` 由后端接收文件再上传 MinIO 的接口。

两种方式的区别：`/oss/info` 是前端拿预签名地址后直传 MinIO，服务端不转发文件内容；`/oss/upload_minio` 是文件先进业务服务再写入 MinIO，会多占一次服务端流量，适合少量或需要后端介入处理的文件。

## 必须对齐的配置

| 配置 | 示例 |
| --- | --- |
| Nacos `biz.oss.endpoint` | `http://<MinIO-IP>:9000` |
| Nacos `biz.oss.resources-url` | `http://<MinIO-IP>:9000/mall4cloud` |
| Nacos `biz.oss.bucket` | `mall4cloud` |
| Nacos `biz.oss.type` | `1` 表示 MinIO |
| 前端 `VITE_APP_RESOURCES_URL` | `http://<MinIO-IP>:9000/mall4cloud` |
| 前端 `VITE_APP_RESOURCES_TYPE` | `1` |

Compose 初始化 SQL 中的 `biz.oss.access-key-secret` 是 `80jpnH4.r5g`，与 `MINIO_ROOT_PASSWORD` 一致。如果 Docker Compose 中 MinIO root 密码改了，Nacos 里的 `biz.oss.access-key-secret` 也要同步。

## 富文本安全

当前平台端和商家端有 `src/directive/rich` 指令，使用 `DOMPurify.sanitize` 后再写入 HTML。商品详情等富文本展示应使用 `v-rich`，不要直接 `v-html` 展示未清洗内容。

后端没有在本轮核对到统一 XSS 过滤器，因此不要把“后端已统一清洗所有输入”写进文档或作为安全假设。

## 检查题

问题：图片上传成功但页面不显示，是否一定是后端上传失败？

答案：不一定。

为什么：上传成功只说明对象写入了 MinIO。展示还依赖桶策略、`resources-url`、前端 `VITE_APP_RESOURCES_URL` 和对象路径拼接。

问题：富文本展示能直接用 `v-html` 吗？

答案：不要直接用，优先使用项目已有 `v-rich`。

为什么：`v-rich` 会先经过 DOMPurify 清洗，直接 `v-html` 会绕开当前前端安全处理。
