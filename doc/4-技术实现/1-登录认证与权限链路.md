# 登录认证与权限链路

权限链路分三层：登录 Token、菜单可见性、按钮/接口权限。二开后台功能时三层都要考虑。

## 登录 Token

1. 前端登录调用 auth 服务。
2. `mall4cloud-auth` 校验账号密码后，通过 `TokenStore.storeAndGetVo` 生成 access token 和 refresh token，并把用户信息写入 Redis。
3. 前端后续请求在 `Authorization` 请求头带 token。
4. 业务服务里的 `AuthFilter` 调 `TokenFeignClient.checkToken(accessToken)` 校验 token。
5. 校验成功后，用户信息放入 `AuthUserContext`。

Token 相关的当前入口：

| 入口 | 作用 |
| --- | --- |
| `LoginController /ua/login` | 登录后清理权限缓存，并返回 token |
| `TokenStore` | 保存、刷新、删除、解析 token |
| `TokenFeignController` | 提供内部 `checkToken` 校验 |
| `AuthFilter` | 每个业务请求进入服务后的 token 和权限校验 |

不需要登录的路径由 `AuthConfigAdapter.excludePathPatterns` 控制。默认放行：

- `/v3/api-docs`
- 内部 Feign 路径
- `/**/ua/**`

## 菜单权限

菜单数据在 `mall4cloud_rbac.menu`。关键字段：

| 字段 | 用途 |
| --- | --- |
| `menu_id` | 菜单 ID |
| `parent_id` | 父菜单 |
| `biz_type` | `1` 商家端，`2` 平台端 |
| `path` | 前端路由路径 |
| `component` | 前端页面组件路径 |
| `title`、`icon` | 侧边栏展示 |

非管理员登录后，前端用 `/mall4cloud_rbac/menu/list_menu_ids` 获取可见菜单 ID，再过滤 `/mall4cloud_rbac/menu/route` 返回的菜单。

## 按钮权限

按钮权限数据在 `mall4cloud_rbac.menu_permission`。关键字段：

| 字段 | 用途 |
| --- | --- |
| `menu_id` | 归属菜单 |
| `permission` | 前端按钮权限编码 |
| `uri` | 后端接口 URI |
| `method` | 请求方法：1 GET、2 POST、3 PUT、4 DELETE |
| `biz_type` | 商家端或平台端 |

角色授权通过 `role_menu` 关联菜单和按钮权限，用户通过 `user_role` 关联角色。

## 后端接口权限

`AuthFilter` 对平台端和商家端用户会调用 `PermissionFeignClient.checkPermission`。当前 `AuthConfigAdapter` 没有全局 `needRbac()` 开关；是否进入 RBAC 校验由用户 `sysType` 决定，平台端和商家端会校验，普通用户端直接放行 RBAC。

`mall4cloud-rbac` 中的实现会：

1. 查询用户拥有的权限编码。
2. 查询当前端所有 `uri + method -> permission` 映射。
3. 用 `AntPathMatcher` 匹配请求 URI 和方法。
4. 如果接口匹配到权限要求，用户必须拥有该权限；如果没有匹配到权限要求，则放行。

这意味着：只加前端按钮权限不够，后端接口权限也要能对应上。

## 新增页面/按钮时要做什么

| 要新增 | 必做 |
| --- | --- |
| 页面菜单 | 新增 `menu`，`component` 对应前端页面路径 |
| 新增按钮 | 新增 `menu_permission`，编码与前端 `v-permission` 一致 |
| 接口保护 | `menu_permission.uri` 和 `method` 对上 Controller 路径和 HTTP 方法 |
| 给角色授权 | 在角色管理中勾选菜单和按钮权限 |

## 排错清单

| 现象 | 答案 |
| --- | --- |
| 菜单不显示 | 用户不是管理员时，先查角色是否勾选该菜单，`list_menu_ids` 是否返回该 `menu_id`。 |
| 按钮不显示 | 查 `v-permission` 编码是否在 `/menu_permission/list` 返回值里。 |
| 按钮显示但接口 401 | 查 `menu_permission.uri`、`method` 是否与真实请求匹配，角色是否拥有该按钮权限。 |
| 管理员可见，普通角色不可见 | 这是正常差异；管理员前端和后端都会走放行逻辑。 |

## 检查题

问题：新增删除按钮，只在 Vue 页面加 `v-permission` 是否足够？

答案：不够。

为什么：`v-permission` 只控制按钮显示，接口访问还要靠 `menu_permission.uri + method` 和角色授权通过后端 `AuthFilter` 校验。
