# 消息、搜索与库存链路

这部分用于理解消息、搜索和库存协作，先阅读再改造。

## RocketMQ

当前代码中可见的消息用途包括：

- 订单取消后解锁库存。
- 支付通知后触发订单和库存后续处理。
- Canal 变更消息驱动搜索索引更新。

注意：提交订单时的初次库存锁定不是 RocketMQ 触发，而是订单服务通过 `SkuStockLockFeignClient.lock` 调商品服务完成。RocketMQ 主要负责延迟取消、延迟解锁和支付成功后的后续通知。

相关入口：

- `mall4cloud-order` 的 RocketMQ 配置和订单监听器。
- `mall4cloud-product` 的库存锁定和解锁监听器。
- `mall4cloud-search` 的 Canal 监听器。

## Elasticsearch

搜索服务依赖 Elasticsearch。索引 mapping 在仓库根目录：

- [product mapping](../../es/product.md)
- [order mapping](../../es/order.md)

商品发布后列表或搜索没有数据时，不要只看商品库，还要检查 Canal、RocketMQ、Elasticsearch 三段是否正常。

Canal 当前监听配置在中间件目录的 `canal/conf` 下：`instance.properties` 指向 MySQL binlog，`canal.properties` 指向 RocketMQ namesrv，topic 是 `canal-topic`。搜索服务通过 `CanalListener` 消费 `RocketMqConstant.CANAL_TOPIC`，再分发到商品、店铺、分类、品牌、订单等监听器。

不要用 `update mall4cloud_product set update_time = now()` 触发同步；当前表结构没有这个表名。需要按 `canal.instance.filter.regex` 里的表名触发，例如 `spu`、`sku`、`sku_stock`、`order` 等。

## 库存锁定

核心表：

- `sku_stock`
- `sku_stock_lock`

下单链路会涉及订单服务和商品服务协作。`sku_stock_lock` 通过 `spu_id + sku_id + order_id` 保证同一订单同一 SKU 的锁定记录唯一。

## 检查题

问题：商品库里有商品，但搜索列表没有数据，最可能只查 MySQL 吗？

答案：不够。

为什么：搜索列表依赖 Elasticsearch 索引，索引更新又可能依赖 Canal 和 RocketMQ。MySQL 有数据只能证明源数据存在。

问题：为什么不要第一个二开功能就选择库存扣减？

答案：因为它横跨订单、商品、消息、事务和库存锁定，风险高。

为什么：学习分层时应先选后台 CRUD，避免一开始就碰交易一致性。
