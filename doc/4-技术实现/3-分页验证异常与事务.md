# 分页、验证、异常与事务

这几个机制是后台 CRUD 二开最常用的公共能力。

## 分页

核心类：

- `PageDTO`
- `PageVO`
- `PageUtil`

Controller 通常接收 `@Valid PageDTO pageDTO`。`PageDTO` 要求 `pageNum`、`pageSize` 非空，并把最大 `pageSize` 限制为 `500`。排序字段会校验字段名，只允许字母、数字、下划线，再把驼峰转下划线。

Service 里常见写法是：

```java
PageUtil.doPage(pageDTO, () -> mapper.list(...))
```

返回给前端的是 `PageVO`：`pages`、`total`、`list`。

常用对象命名按当前代码理解：

| 名称 | 当前用法 |
| --- | --- |
| `Model`/PO | 与数据库表结构对应，放在 `model` 目录 |
| `DTO` | 前端传入参数，例如 `PageDTO`、`SpuDTO` |
| `VO` | 返回给前端的数据，例如 `PageVO` |
| `BO` | 服务内部或跨服务传递的业务对象 |

当前前端是 Vue 3，不使用 Vue 2 的 `.sync` 写法；页面交互以现有 Vue 3 页面和组件为准。

## 验证

DTO 字段使用 Jakarta Validation 注解，例如 `@NotNull`、`@NotBlank`。Controller 参数上加 `@Valid` 后，验证失败会进入统一异常处理。

## 统一异常

`DefaultExceptionHandlerConfig` 处理：

| 异常 | 返回 |
| --- | --- |
| `MethodArgumentNotValidException`、`BindException` | `METHOD_ARGUMENT_NOT_VALID`，并返回字段错误列表 |
| `HttpMessageNotReadableException` | `HTTP_MESSAGE_NOT_READABLE` |
| `Mall4cloudException` | 使用异常自带枚举或直接显示消息 |
| 其他 `Exception` | `EXCEPTION`，并在存在 Seata XID 时尝试回滚全局事务 |

项目约定：请求到达服务并被业务处理后，大多数失败也返回 HTTP 200，前端按业务 `code` 判断。

## 分布式事务

跨服务写操作参考已有 `@GlobalTransactional`。典型位置包括订单、商品、平台用户、商家用户等服务。不要只在本地方法上加事务就认为跨服务一致性已经解决。

## Redis 锁

公共缓存模块提供 `RedisLockRegistry`，锁前缀为 `spring-cloud`。需要并发控制时优先看是否已有业务模式可复用。

## 检查题

问题：列表接口返回值应该是裸 `List<T>` 还是 `PageVO<T>`？

答案：后台分页列表应该返回 `PageVO<T>`。

为什么：当前分页公共模型包含 `pages`、`total`、`list`，前端分页组件依赖这些字段。

问题：`pageSize=10000` 会照单全查吗？

答案：不会按 10000 查，`PageDTO` 会把超过 `500` 的分页大小限制为 `500`。

为什么：`PageDTO.setPageSize` 中有 `MAX_PAGE_SIZE = 500` 的保护。
