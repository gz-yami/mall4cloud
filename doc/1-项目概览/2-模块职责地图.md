# 模块职责地图

本页帮助你先知道“改哪里”，再去看细节。

## 后端服务

| 模块 | 职责 | 常见二开入口 |
| --- | --- | --- |
| `mall4cloud-gateway` | 网关，端口 `8000`，按 Nacos 路由把 `/mall4cloud_xxx/**` 转给服务 | 调整跨域、路由、统一入口 |
| `mall4cloud-auth` | 登录、Token 发放和校验 | 登录方式、账号体系 |
| `mall4cloud-rbac` | 菜单、角色、按钮权限、接口权限 | 新增菜单和按钮权限 |
| `mall4cloud-platform` | 平台端后台业务 | 平台用户、系统配置、平台运营功能 |
| `mall4cloud-multishop` | 商家端后台业务 | 店铺用户、店铺装修、商家运营功能 |
| `mall4cloud-product` | 商品、分类、品牌、SKU、库存 | 商品管理和库存相关功能 |
| `mall4cloud-order` | 订单、订单项、收货地址、发货 | 订单查询和发货，先阅读再改造 |
| `mall4cloud-payment` | 支付信息和支付通知 | 支付接入，先阅读再改造 |
| `mall4cloud-biz` | 文件上传、附件等通用业务 | MinIO 上传、附件管理 |
| `mall4cloud-search` | Elasticsearch 搜索、Canal 消息消费 | 商品/订单索引同步 |
| `mall4cloud-user` | 用户、地址、区域 | 用户资料和收货地址 |
| `mall4cloud-leaf` | 分布式 ID | 新表主键生成策略 |
| `mall4cloud-api` | 服务间 Feign 接口 | 跨服务调用契约 |
| `mall4cloud-common` | 公共响应、异常、分页、安全、缓存、数据库、消息 | 新功能复用基础能力 |

## 前端工程

| 工程 | 角色 | 怎么读 |
| --- | --- | --- |
| `front-end/mall4cloud-platform` | 平台端管理后台 | 重点读登录、动态菜单、后台列表页模式 |
| `front-end/mall4cloud-multishop` | 商家端管理后台 | 重点读商家端菜单和同类 CRUD 页面 |
| `front-end/mall4cloud-uniapp` | 用户端 | 先核对接口地址，后台跑通后再看页面 |

平台端和商家端都是 Vue 3 + Vite + Pinia + Element Plus，环境变量使用 `VITE_APP_*`。

## 数据库脚本

| 目录/文件 | 用途 |
| --- | --- |
| `db/*.sql` | 当前库表结构和初始数据，是数据模型文档的主要依据 |
| `doc/8-部署运维/2-中间件-docker-compose/mysql/initdb/*.sql` | Docker Compose 启动中间件时导入 Nacos、Seata 等配置 |
| `es/product.md`、`es/order.md` | Elasticsearch 索引 mapping |

## 请求链路

管理后台请求 `VITE_APP_BASE_API`，默认指向网关 `8000`。网关读取 Nacos 中的路由配置，例如 `/mall4cloud_rbac/**` 转到 `mall4cloud-rbac`。业务服务再通过 MySQL、Redis、RocketMQ、Seata、MinIO、Elasticsearch 等中间件完成各自职责。

## 练习：我应该改哪个模块

问题：要新增一个“公告管理”后台功能，第一步应该去订单服务吗？

答案：不应该。公告管理更像平台端运营能力。优先在 `mall4cloud-platform` 增加后端接口，在 `front-end/mall4cloud-platform` 增加页面，在 `mall4cloud-rbac` 的菜单和按钮权限中补权限。

为什么：订单服务涉及交易状态和库存协作，不适合作为学习项目分层的第一个改动点。
