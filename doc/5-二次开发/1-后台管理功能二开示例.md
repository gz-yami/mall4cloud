# 后台管理功能二开示例

本示例选择“平台公告管理”作为学习题材。它是后台 CRUD，不直接触碰订单、支付、库存、结算。本页只写教程和检查清单，不在仓库中新增业务代码。

## 目标

做一个平台端公告管理：

- 列表查询公告。
- 新增公告。
- 编辑公告。
- 删除或禁用公告。
- 平台端菜单可见。
- 普通角色只有授权后才能看到按钮和访问接口。

## 后端步骤

| 步骤 | 应该做什么 | 验收 |
| --- | --- | --- |
| 建表 | 在平台业务库新增公告表，包含主键、标题、内容、状态、排序、创建/更新时间 | 表能初始化，字段含义明确 |
| Model | 新增与表对应的 Model | 字段类型和数据库一致 |
| DTO | 新增保存/更新入参 DTO | 必填字段有验证注解 |
| VO | 新增列表和详情返回 VO | 不把不需要的内部字段暴露给前端 |
| Mapper/XML | 写分页列表、详情、新增、更新、删除或改状态 SQL | SQL 字段名和表字段一致 |
| Service | 封装业务逻辑 | Controller 不直接写 SQL |
| Controller | 提供 page、get、save、update、delete 接口 | 返回 `ServerResponseEntity` |
| 分页 | 列表使用 `PageDTO` 和 `PageVO` | 前端能拿到 `total` 和 `list` |

接口路径放在平台端语义下，例如 `/platform/notice`，或采用现有平台模块风格。落地前先参考 `HotSearchController`、`IndexImgController`、`SysUserController` 等后台接口。

## 前端步骤

| 步骤 | 应该做什么 | 验收 |
| --- | --- | --- |
| API | 在 `src/api` 下新增公告接口文件 | 请求都走网关前缀 |
| 列表页 | 在 `src/views/modules/platform/notice` 下新增页面 | 能查询、分页、重置 |
| 表单弹窗 | 新增或复用 add-or-update 组件 | 新增和编辑共用校验 |
| 按钮权限 | 新增、编辑、删除按钮加 `v-permission` | 无权限用户按钮不可见 |
| 环境变量 | 不新增 `VUE_APP_*` | 仍使用 `VITE_APP_BASE_API` |

## 菜单和按钮权限

需要在 rbac 中配置：

| 类型 | 示例 |
| --- | --- |
| 菜单 | `menu.path = notice`，`menu.component = platform/notice` |
| 新增按钮 | `platform:notice:save`，POST `/platform/notice` |
| 编辑按钮 | `platform:notice:update`，PUT `/platform/notice` |
| 删除按钮 | `platform:notice:delete`，DELETE `/platform/notice` |

编码只是示例，落地命名要和现有平台端权限风格保持一致。

## 验收清单

- 平台管理员能看到公告菜单。
- 普通角色未授权时看不到菜单。
- 普通角色授权菜单后能看到菜单。
- 未授权按钮权限时，新增/编辑/删除按钮不显示。
- 直接请求未授权接口会被后端拒绝。
- 列表分页返回 `PageVO`。
- 表单必填项缺失时返回统一验证错误。

## 检查题

问题：这个示例为什么选公告管理，而不是订单退款？

答案：公告管理是后台 CRUD，能覆盖分层、分页、表单、菜单和按钮权限；订单退款涉及交易一致性，不适合作为第一课。

为什么：学习的目标是先掌握项目模式，再进入高风险业务。
