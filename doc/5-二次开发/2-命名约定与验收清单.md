# 命名约定与验收清单

新功能要像项目原生功能，而不是单独长出来的一套风格。

## 命名约定

| 对象 | 约定 |
| --- | --- |
| 数据库表 | 下划线命名，如 `notice`、`notice_item` |
| 主键 | `xxx_id` 或同业务已有风格 |
| Java 类 | `NoticeController`、`NoticeService`、`NoticeMapper`、`NoticeDTO`、`NoticeVO` |
| 前端目录 | 放到 `src/views/modules/<业务域>/<功能>` |
| API 文件 | 放到 `src/api/<业务域>` |
| 权限编码 | `<业务域>:<功能>:<动作>` |
| 接口路径 | 跟当前 Controller 风格一致，不混用多个前缀 |

## 后端验收

- Controller 返回 `ServerResponseEntity`。
- 新增/更新入参使用 DTO，并对必填字段加验证。
- 列表使用 `PageDTO`、`PageVO`。
- Service 负责业务编排，Mapper 只做数据访问。
- 涉及跨服务写操作时评估是否需要 `@GlobalTransactional`。
- 接口路径和 HTTP 方法能与 `menu_permission` 对上。

## 前端验收

- 请求走 `VITE_APP_BASE_API` 指向网关。
- 页面路径能被 `menu.component` 动态加载。
- 按钮使用 `v-permission`。
- 列表、表单、删除确认、加载态、错误提示与现有页面风格一致。
- 不新增 `VUE_APP_*` 或 Vue 2/Vue CLI 配置。

## 权限验收

- 管理员可见全部菜单和按钮。
- 普通角色只在授权后可见菜单。
- 普通角色只在授权后可见按钮。
- 直接调用未授权接口也不能成功。

## 检查题

问题：权限编码写成 `notice:add` 可以吗？

答案：不要，除非当前业务域已有这种风格。

为什么：项目现有编码多是 `<业务域>:<功能>:<动作>`，例如 `product:category:save`。统一风格方便维护和排查。
