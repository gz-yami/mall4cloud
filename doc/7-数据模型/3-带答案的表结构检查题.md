# 带答案的表结构检查题

这些题不是考试，是帮助读者确认自己已经读懂关键关系。

## 权限

问题：一个普通平台用户登录后，为什么可能看不到某个菜单？

答案：因为用户不是管理员时，前端会根据 `/menu/list_menu_ids` 返回的菜单 ID 过滤动态路由；如果角色没有关联该菜单，菜单就不会显示。

为什么：用户通过 `user_role` 找到角色，角色通过 `role_menu.menu_id` 拥有菜单，前端再用这些菜单 ID 过滤 `menu` 路由。

问题：按钮显示和接口放行是不是同一件事？

答案：不是。

为什么：按钮显示由前端 `v-permission` 判断 `menu_permission.permission`；接口放行由后端根据 `menu_permission.uri + method` 和用户权限判断。

## 商品

问题：SPU 和 SKU 的区别是什么？

答案：SPU 是商品主信息，SKU 是具体可销售规格。

为什么：`spu` 保存商品名称、店铺、分类、上下架状态；`sku` 保存 `spu_id`、规格属性、价格和 SKU 状态。

问题：库存为什么不只放在 `sku` 表？

答案：库存独立放在 `sku_stock`，锁定记录放在 `sku_stock_lock`。

为什么：库存会被下单、取消、支付等链路频繁改变，独立表和锁定表便于处理可售库存、实际库存、锁定库存和订单锁定记录。

## 订单

问题：订单明细为什么要保存商品名称、SKU 名称和价格？

答案：因为订单需要保留下单时的商品快照。

为什么：商品后续可能改名、改价或下架，`order_item` 仍要展示和结算下单时的信息。

问题：订单表里的 `total` 和订单项里的 `spu_total_amount` 可以直接改成小数吗？

答案：不要。

为什么：当前金额字段以整数保存，随意改成小数会影响支付、结算、前后端展示和已有数据。

## 二开

问题：新增“公告管理”表是否一定要有 `shop_id`？

答案：不一定，要看公告是平台全局还是店铺独立。

为什么：平台全局公告可以归平台端管理；店铺独立公告才需要 `shop_id` 或租户维度。建表前先明确数据归属。

问题：新表一定要有 `undo_log` 吗？

答案：新业务表本身不需要 `undo_log` 字段；参与 Seata 分布式事务的数据库需要有独立的 `undo_log` 表。

为什么：`undo_log` 是 Seata AT 模式使用的事务回滚表，不是每张业务表的字段。
